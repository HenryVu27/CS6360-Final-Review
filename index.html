DBMS Fundamentals

Key Definitions

Data: Raw facts that can be recorded and have implicit meaning
Information: Processed data
Database: Collection of data central to some enterprise (persistent)
Mini-world: Part of real world about which data is stored
Metadata: Description of the database stored in DBMS catalog
DBMS: Software package that manages a database
Database System: DBMS software + data (sometimes includes applications)
Schema (Intension): Description/structure of the database, changes infrequently
Instance (Extension): Actual data at a particular moment, changes with every update

What a DBMS Provides

  • High-level access language (e.g., SQL)

  • Special operations not supported

Database Models Comparison

Hierarchical Model

Structure: Tree (parent-child)

Pros: Simplicity, security, integrity, efficiency

Cons: Complex implementation, lacks structural independence

Network Model

Structure: Child can have multiple parents

Pros: Handles more relationship types, data access flexibility

Cons: System complexity, lacks structural independence

Relational Model

Structure: Data as tables (relations)

Pros: Structural independence, simplicity, SQL, powerful DBMS

Cons: Hardware/software overhead, potential poor design

DBMS Languages

DDL (Data Definition Language)

  • Define conceptual schema
  • Define internal and external schemas
  • Result: catalog (metadata stored in file)

Examples: CREATE, ALTER, DROP

DML (Data Manipulation Language)

  • Specify retrievals and updates
  • High-level (nonprocedural): Describe WHAT data is needed
  • Low-level (procedural): Describe WHAT and HOW

Examples: SELECT, INSERT, UPDATE, DELETE

SDL (Storage Definition Language)

  • Directives that influence physical schema
  • Specifies storage structures

Types of Relational DMLs

  • Relational Algebra: ISBL
  • Tuple Calculus: SQL
  • Domain Calculus: QBE

A DML with same "retrieval power" as relational algebra = RELATIONALLY COMPLETE

Integrity Constraints

Three Main Types

Key Constraints

Superkey: Set of attributes where no two tuples have same value

Key (Candidate Key): MINIMAL superkey

Entity Integrity

Primary key attributes CANNOT be NULL

Referential Integrity

Conditions:

  • Dom(R1.FK) = Dom(R2.PK)

Required for: UNION, INTERSECTION, SET DIFFERENCE

ER-to-Relational Mapping

Basic Mapping Rules

Entity → Table
Attribute → Column
Many-to-many relationships → New table
One-to-one and one-to-many → Key propagation (foreign key)
Relationship with attributes → Map to a table

Mapping Keys

  • Strong entities: Use entity's key as table's primary key
  • Weak entities: Combine entity's partial key + strong entity's key
  • Relationship tables: Use keys from both entity types as composite key

EER Mapping (Generalization/Specialization)

Given superclass S with attributes (s1, s2, s3) and subclasses A(a) and B(b):

Alternative 1: Multiple Tables

Superclass + Subclass tables

S(s1, s2, s3)     -- s1 is PK
A(s1, a)          -- s1 is FK to S
B(s1, b)          -- s1 is FK to S

Alternative 2: Only Subclass Tables

All superclass attributes + subclass attributes

A(s1, s2, s3, a)
B(s1, s2, s3, b)

Alternative 3: Single Table

If DISJOINT:

C(s1, s2, s3, a, b)
-- s3 is discriminator attribute

If OVERLAPPING:

C(s1, s2, s3, aflag, a, bflag, b)
-- flags indicate which subclass

Participation Constraints in EER Mapping

  • Total participation: At least one flag must be true
  • Non-total participation: One or both flags may be false

Ternary and Higher-Degree Relationships

Degree of Relationship

  • Binary: Connects 2 entity types (most common)
  • Ternary: Connects 3 entity types
  • N-ary: Connects n entity types

Important: N-ary relationship is NOT equivalent to n binary relationships!

When to Use Ternary Relationships

  • When relationship involves three entity types simultaneously
  • Binary relationships cannot capture the same information
  • When specific combination of all three matters

Example: SUPPLY(Supplier, Part, Project) with Quantity attribute

Ternary vs Binary

  • Cannot always decompose ternary into binaries
  • Information may be lost
  • Use ternary when specific combination of all three matters

Constraints on Ternary

m:n:p notation on three arms is AMBIGUOUS (does not fully specify constraints)

Use (min,max) notation instead for precision

Specifies how many relationship instances each entity participates in

Recursive Relationship

  • Same entity type participates twice with different roles
  • Example: EMPLOYEE supervises EMPLOYEE (supervisor/supervisee roles)
  • Must label roles on diagram edges

Weak Entity Types

Definition: Entity that does NOT have its own key attribute. Depends on "owner" or "identifying" entity type.

Identification

Full Key = Partial Key + Owner's Primary Key

Example: DEPENDENT identified by Name + Employee's SSN

Notation

  • Weak entity: DOUBLE RECTANGLE
  • Identifying relationship: DOUBLE DIAMOND
  • Partial key: DASHED UNDERLINE

Requirements

  • Identifying relationship is 1:N from owner to weak entity (or 1:1)
  • Each weak entity has exactly one owner; owner can have many weak entities
  • Total participation from weak entity side (exactly one owner)

Subclass Constraints

(d) Disjoint: Entity in AT MOST ONE subclass
(o) Overlapping: Entity in MULTIPLE subclasses allowed

Completeness

Total (double line): Entity MUST be in some subclass

Inheritance

  • Subclass inherits ALL attributes from superclass
  • Subclass inherits ALL relationships from superclass
  • Subclass can have its own specific attributes/relationships

Subclass can have multiple superclasses (multiple inheritance)

Forms a lattice rather than simple hierarchy

Design Guidelines

1. Avoid Redundancy

  • Don't say same thing more than once
  • Wastes space and causes inconsistency

BAD

Beer entity with ManufacturerAddress (repeats for each beer)

GOOD

Separate Manufacturer entity with address

2. Enforce Constraints in Schema

  • Don't rely on data to follow assumptions
  • Schema should prevent violations
  • Example: If only one instructor per course, enforce it

3. Entity Set vs Attribute Decision

Make entity set only if:

  • Has non-key attributes OR relationships with multiple entities
  • Is the "many" side in many-one relationship

Otherwise use attribute.

4. Don't Overuse Weak Entities

  • In practice, create unique IDs (SSN, VIN, etc.)
  • Only use weak entities when truly dependent

ER Model Limitations

  • No relationship between entity type and relationship type
  • No relationship between entity type and collection of entity types
  • No constraints (exclusion, co-existence) among relationship types
  • Note: UML and NIAM models allow these

Normalization

Informal Design Guidelines

  1. Each tuple should represent one entity or relationship instance
    • Don't mix attributes of different entities (EMPLOYEEs, DEPARTMENTs, PROJECTs)
    • Only use foreign keys to refer to other entities
  2. Design schemas that don't suffer from insertion, deletion, and update anomalies
  3. Design relations to minimize NULL values (frequently NULL attributes should be in separate relations)

Update Anomalies

Redundancy leads to three types of anomalies:

Update Anomaly

A change must be made in multiple places

Deletion Anomaly

Deleting data may cause loss of other information

Insertion Anomaly

Cannot insert data without providing other unrelated data

Functional Dependency (FD)

X ‚Üí Y means: for every pair of tuples t and s, if they agree on all attributes in X, they must agree on all attributes in Y

Examples

  • SSN ‚Üí Name, Address
  • Address ‚Üí ZipCode
  • Author, Title ‚Üí PublDate

Armstrong's Axioms

  1. Reflexivity: If Y ‚äÜ X, then X ‚Üí Y (trivial FD)
  2. Augmentation: If X ‚Üí Y, then XZ ‚Üí YZ
  3. Transitivity: If X ‚Üí Y and Y ‚Üí Z, then X ‚Üí Z

Derived Rules

  • Union: If X ‚Üí Y and X ‚Üí Z, then X ‚Üí YZ
  • Decomposition: If X ‚Üí YZ, then X ‚Üí Y and X ‚Üí Z

Attribute Closure

X+ (attribute closure of X with respect to F) = set of all attributes A such that X ‚Üí A

Algorithm to Compute X+

closure := X
repeat
    old := closure
    for each FD Z ‚Üí V in F:
        if Z ‚äÜ closure and V ‚àâ closure:
            closure := closure ‚à™ V
until old = closure

Use of Attribute Closure

To check if X ‚Üí Y is entailed by F, verify if Y ‚äÜ X+

Minimal Cover

A minimal cover U of F satisfies:

  • U is equivalent to F (same closure)
  • All FDs have form X ‚Üí A (single attribute on RHS)
  • Cannot make U smaller by deleting FD or attribute

Computing Minimal Cover

  1. Decompose RHS to single attributes
  2. Remove unnecessary LHS attributes (if X ‚Üí A and XY ‚Üí A both hold, Y is unnecessary)
  3. Remove unnecessary FDs (if F - {f} entails f, then f is unnecessary)

Key Concepts

  • Superkey: Set of attributes that uniquely identifies tuples
  • Candidate Key: Minimal superkey (no proper subset is a superkey)
  • Primary Key: Chosen candidate key
  • Prime Attribute: Attribute that is part of some candidate key
  • Non-prime Attribute: Attribute not part of any candidate key
  • Full Functional Dependency: X ‚Üí Y where no attribute can be removed from X
  • Partial Dependency: X ‚Üí Y where some attribute can be removed from X
  • Transitive Dependency: X ‚Üí Y ‚Üí Z (Z depends on X through Y)

1NF

1NF (First Normal Form)

  • No composite attributes
  • No multivalued attributes
  • No nested relations
  • All attribute values must be atomic

2NF (Second Normal Form)

  • Must be in 1NF
  • No partial dependencies
  • Every non-prime attribute is fully functionally dependent on every key

Bad: {SSN, PNUMBER} ‚Üí ENAME is partial because SSN ‚Üí ENAME alone holds

3NF (Third Normal Form)

For every FD X ‚Üí A:

  • A ‚äÜ X (trivial), OR
  • X is a superkey, OR
  • A is part of some key (prime attribute)

3NF allows some redundancy but guarantees dependency-preserving decomposition

BCNF (Boyce-Codd Normal Form)

For every FD X ‚Üí Y:

  • Y ‚äÜ X (trivial), OR
  • X is a superkey

BCNF eliminates redundancy but decomposition may not be dependency-preserving

4NF (Fourth Normal Form)

For every non-trivial multi-valued dependency:

  • X ‚à© Y is a superkey of R

4NF implies BCNF

Decomposition

Goal: Eliminate redundancy by decomposing into relations in higher normal form

Requirements for Good Decomposition

1. Lossless

Original relation can be reconstructed

r = r1 ‚ãà r2 ‚ãà ... ‚ãà rn

2. Dependency-preserving

All FDs can be checked without joining tables

F+ = (F1 ‚à™ F2)+

Test for Lossless Decomposition (Binary)

Decomposition of R into R1 and R2 is lossless if and only if:

  • R1 ‚à© R2 ‚Üí R1 is in F+, OR
  • R1 ‚à© R2 ‚Üí R2 is in F+

The common attributes must be a key of at least one decomposed relation

BCNF Decomposition Algorithm

while there exists S in Decomp not in BCNF: find X ‚Üí Y that violates BCNF (X is not superkey) replace S with: S1 = (XY, F1) S2 = (S - (Y-X), F2)

Properties: Always lossless, may NOT be dependency-preserving

3NF Synthesis Algorithm

  1. Compute minimal cover U of F
  2. Partition U by LHS (group FDs with same left side)
  3. For each group, create schema Ri with attributes from that group's FDs
  4. If no Ri contains a key of R, add R0 = (key attributes, {})

Properties: Lossless, dependency-preserving, in 3NF

Denormalization

Tradeoff: Intentionally introduce redundancy to improve query performance

  • Avoids expensive joins for frequently-run queries
  • Sacrifices BCNF/3NF properties
  • Requires careful application-level management of redundancy

SQL Queries

Basic SELECT Structure

SELECT <attribute list>
FROM <table list>
WHERE <condition>
GROUP BY <grouping columns>
HAVING <group conditions>
ORDER BY <sorting specifications>

Key Points

  • SQL relation is a MULTI-SET (allows duplicates)
  • Use DISTINCT to eliminate duplicates
  • * selects all columns

Pattern Matching

WHERE CourseNo LIKE 'IS%'   -- % = any characters
WHERE Name LIKE 'J_hn'      -- _ = single character

NULL Testing

WHERE FacSSN IS NULL
WHERE FacSSN IS NOT NULL

Query Evaluation Order

  1. FROM - Cartesian product / joins
  2. WHERE - Filter rows
  3. GROUP BY - Sort and group rows
  4. Compute aggregates, reduce each group to one row
  5. HAVING - Filter groups
  6. SELECT - Project columns
  7. ORDER BY - Sort results

Self-Join

SELECT T1.ProfId
FROM Teaching T1, Teaching T2
WHERE T1.ProfId = T2.ProfId
  AND T1.Semester = T2.Semester
  AND T1.CrsCode <> T2.CrsCode;

Outer Joins

LEFT OUTER JOIN

All rows from left + matching from right

FROM T1 LEFT JOIN T2 ON condition

RIGHT OUTER JOIN

All rows from right + matching from left

FROM T1 RIGHT JOIN T2 ON condition

FULL OUTER JOIN

All rows from both tables

FROM T1 FULL JOIN T2 ON condition

Aggregate Functions

Functions

COUNT(*) - Count all rows
COUNT(col) - Count non-null values
SUM(col) - Sum of values
AVG(col) - Average of values
MAX(col) - Maximum value
MIN(col) - Minimum value

GROUP BY

SELECT T.StudId, AVG(T.Grade), COUNT(*)
FROM Transcript T
GROUP BY T.StudId;

HAVING (Filter Groups)

SELECT T.StudId, AVG(T.Grade) AS CumGpa
FROM Transcript T
WHERE T.CrsCode LIKE 'CS%'
GROUP BY T.StudId
HAVING AVG(T.Grade) > 3.5;

Important Rules

  • Cannot mix aggregates with non-aggregate columns without GROUP BY
  • Aggregates cannot be used in WHERE clause
  • GROUP BY must list ALL non-aggregate columns from SELECT
  • WHERE: row conditions (before grouping)
  • HAVING: group conditions (after grouping)

Nested Queries (Subqueries)

Type I - Non-Correlated (Independent)

  • Subquery executes ONCE
  • No reference to outer query
  • Used with IN, NOT IN, comparison operators
SELECT C.CrsName
FROM Course C
WHERE C.CrsCode NOT IN
    (SELECT T.CrsCode FROM Teaching T WHERE T.Sem = 'S2000');

Type II - Correlated

  • Subquery executes for EACH ROW of outer query
  • References outer query variables
  • Used with EXISTS, NOT EXISTS
SELECT P.Name
FROM Professor P
WHERE EXISTS (
    SELECT * FROM Teaching T
    WHERE T.ProfId = P.Id    -- correlation to outer query
    AND T.Semester = 'F2000');

Division in SQL

Find professors who teach courses in ALL departments

SELECT P.Id
FROM Professor P
WHERE NOT EXISTS
    (SELECT D.DeptId FROM Department D
     EXCEPT
     SELECT C.DeptId
     FROM Teaching T, Course C
     WHERE T.ProfId = P.Id AND T.CrsCode = C.CrsCode);

Nested Queries in FROM Clause

Used for nested aggregates (aggregate of aggregates) and multiple independent calculations:

SELECT T.CourseNo, COUNT(*) AS NumOfferings, AVG(T.EnrollCount)
FROM
    (SELECT Course.CourseNo, Offering.OfferNo,
            COUNT(*) AS EnrollCount
     FROM Offering, Enrollment, Course
     WHERE Offering.OfferNo = Enrollment.OfferNo
       AND Course.CourseNo = Offering.CourseNo
     GROUP BY Course.CourseNo, Offering.OfferNo) T
GROUP BY T.CourseNo;

Views

Definition

A virtual table (not physically stored) - result of a SELECT statement, materialized when used

CREATE VIEW CumGpa (StudId, Cum) AS
    SELECT T.StudId, AVG(T.Grade)
    FROM Transcript T
    GROUP BY T.StudId;

Benefits of Views

  • Access Control: Limit visibility to certain columns/rows
  • Customization: Simplify complex database for users
  • Reusability: Can be used in multiple queries

Updating Views

Updatable view restrictions:

  • No Cartesian product in FROM clause
  • No aggregates, GROUP BY, HAVING
  • Single base table

Problems with View Updates

  • Projected-out columns get NULL or DEFAULT
  • New tuples may not appear in view (use WITH CHECK OPTION)
  • Ambiguous updates on joins

NOT

  • NOT TRUE = FALSE
  • NOT FALSE = TRUE
  • NOT UNKNOWN = UNKNOWN

NULL Behavior

  • Comparison with NULL yields UNKNOWN
  • Rows with UNKNOWN in WHERE are discarded
  • Arithmetic with NULL yields NULL
  • COUNT(*) counts all rows; COUNT(column) ignores nulls
  • Other aggregates (SUM, AVG, etc.) ignore nulls
  • In GROUP BY, NULL values are grouped together

Methods to Find "Rows NOT in Another Set"

  1. Type II nested query with NOT EXISTS
  2. Type I nested query with NOT IN
  3. One-sided outer join with IS NULL
  4. MINUS/EXCEPT operator

Transaction Processing

Transaction Basics

Transaction: A logical unit of database processing that includes one or more access operations (read/retrieval, write/insert/update, delete).

Key Characteristics

  • Consistent on-line database despite failures and interleavings
  • Small work units (e.g., transfer, deposit, withdraw)
  • Throughput: # transactions per second (TPS)
  • Concurrency: # units overlapping a given unit

Transaction Operations

  • read_item(X) - read database item X into program variable
  • write_item(X) - write to database
  • begin/end transaction - define transaction boundaries
  • commit - make changes permanent
  • abort (rollback) - undo all changes

ACID Properties

A - Atomicity

All actions performed or none (all-or-nothing)

C - Consistency

Moves database from one consistent state to another

I - Isolation

Transaction's updates not visible to others before commit

D - Durability

Committed changes to DB are permanent

Transaction States

Active ‚Üí Partially Committed ‚Üí Committed

Active ‚Üí Failed ‚Üí Terminated

Concurrency Problems

  • Lost Update: Two transactions update same item, one update lost
  • Dirty Read: Reading uncommitted data that may be rolled back
  • Unrepeatable Read: Same query returns different results in same transaction
  • Incorrect Summary: Aggregate computed while another transaction updating

Recovery

Goal: Either all actions in a transaction are completed or has no effect on database or other transactions

Failure Types

  • Catastrophic failure - complete system crash
  • System failure - HW/disk/OS failure
  • Transaction error/exception - programming errors
  • Rollback due to concurrency control - deadlock resolution

System Log (Journal)

Keeps track of all transaction operations affecting database item values.

Log Record Types

  • [start_transaction, T] - T has started
  • [write_item, T, X, old_value, new_value] - T changed X
  • [read_item, T, X] - T read X
  • [commit, T] - T completed successfully
  • [abort, T] - T has been aborted

Recovery Operations

UNDO

  • Applies to single operation
  • Trace BACKWARD through log
  • Reset items to OLD values

REDO

  • Ensures committed operations applied
  • Trace FORWARD through log
  • Set items to NEW values

Commit Point

When all transaction operations accessing DB executed successfully AND all effects recorded in the log.

Force Writing: Before commit point, any unwritten log portion must be written to disk

Schedules

Schedule S(T1,...,Tn): An ordering of operations such that the order of operations of Ti in S equals the order of operations in Ti

Ordering can be total or partial

Notation

  • ri(X) - read item X by Ti
  • wi(X) - write item X by Ti
  • ci - commit Ti
  • ai - abort Ti

Committed Projection

C(S) = operations from committed transactions only

Number of Schedules Formula

#schedules S(T1,...,Tn) = N! / (N1! √ó N2! √ó ... √ó Nn!)

where Ni = number of operations in Ti, N = N1 + N2 + ... + Nn

Recoverability Properties

Strict

No concurrent read/write on any data item

wj(X) < cj < ri(X) AND wj(X) < cj < wi(X)

‚äÇ

Cascadeless

No dirty read - If Ti reads from Tj, then cj < ri(X)

‚äÇ

Recoverable

If Ti reads from Tj, then cj < ci

Serializability

Why important: Characterization of "correctness" - legal interleavings

Types of Schedules

  • Serial: No interleaving; all actions of Ti together in S
  • Non-serial: Interleaving exists; for some Ti, actions not together
  • Serializable: Equivalent to some serial schedule of same transactions

Equivalence Types

Result Equivalent

Produce same final state of DB

NOT sufficient for correctness!

Conflict Equivalent

Order of any two conflicting operations is identical

This is the standard for correctness

Conflicting Operations

On same item X, different transactions:

Wi(X) ... Wj(X) CONFLICT
Wi(X) ... Rj(X) CONFLICT
Ri(X) ... Wj(X) CONFLICT
Ri(X) ... Rj(X) NOT conflict

Precedence Graph Algorithm

  1. Look at only ReadItem(X) and WriteItem(X) operations
  2. Construct precedence graph:
    • Nodes = transactions T1, T2, ..., Tn
    • Edges = conflicts (Ti ‚Üí Tj if conflict and Ti's operation is first)
  3. Check for cycles

NO CYCLE in precedence graph ‚ü∫ Schedule is CONFLICT SERIALIZABLE

If serializable, topological sort gives equivalent serial order

Global Serializability (Distributed Databases)

For distributed transactions:

  • Must ensure subtransactions are serializable at each site
  • Must also ensure global serialization over all sites
